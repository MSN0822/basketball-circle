# ぶらんかーず 参加管理システム 運営向け仕様書

最終改訂: 2026-07-21（実装に追従して随時更新）  
対象: 運営メンバー向けの機能説明、事前確認、当日運用

> 本ドキュメントは、現在のシステム実装に合わせて維持する「生きた仕様書」です。仕様変更があった都度、内容を更新してください。日付ごとの確認チェックリストは設けません。

## 1. システムの目的

「ぶらんかーず」のイベント参加申請を、参加者本人と運営の双方が確認・管理できるようにするためのWebシステムです。

主な目的は次の通りです。

- イベントごとの参加者数を管理する
- 参加者本人が参加申請・キャンセルできるようにする
- 参加者が友達を代理で追加できるようにする
- 定員到達時に追加申請を止め、参加者が再開閾値を下回ったら自動で受付再開する
- 運営側でイベント作成、編集、公開、締切、参加者確認を行う

## 2. 利用者区分

### 参加者

通常のユーザです。メールアドレスとパスワードで登録・ログインします。

できること:

- イベント一覧を見る
- イベント詳細を見る
- 自分の参加申請をする
- 自分の参加をキャンセルする
- 友達を代理で追加する
- 自分が追加した友達をキャンセルする
- ニックネームを変更する

### 運営

管理者画面を使うユーザです。管理者パスワードでログインします。

できること:

- イベントを作成する
- イベントを編集する
- イベントを下書き保存する
- イベントを公開する（即時公開、または公開予約日時による自動公開）
- イベントを締め切る（手動）
- イベントを再開する（手動）
- イベントをアーカイブ化する（終了後の非表示移行）
- アーカイブ済みイベントを閲覧する
- 参加者一覧を確認する
- 参加者を強制キャンセルする

## 3. 参加者登録仕様

新規登録時に入力する項目は次の通りです。

- 姓（必須）
- 名（必須）
- ニックネーム（任意）
- メールアドレス（必須）
- パスワード（6文字以上、必須）

### メール確認（6桁の確認コード）

新規登録ではメール確認が必須です。登録ボタンを押すと、入力したメールアドレスに6桁の確認コードが送られ、「確認コードを送信しました」と表示されます。「コード入力へ進む」を押すとコード入力画面に切り替わります。

6桁の確認コードを入力して登録を完了すると、会員情報（members 行）が作成され、トップページへ移動します。

確認コードを入力しないまま放置すると、コードは無効になります。その場合は再度登録し直すか、ログイン画面からやり直してください。

### メール送信上限について

Supabase の無料枠では、確認メールの送信は1時間あたり最大2通に制限されています。上限に達した場合は、「現在、確認メールの送信上限に達しています。お手数ですが、1時間ほど空けてからもう一度登録してください。」といった内容のエラーメッセージが表示されます。短時間に複数回の登録テストを行うと、この上限に達することがあるため注意してください。

メール確認方式は運用判断で変更できます（独自 SMTP への移行で上限を解消する案／メール確認自体を止める案）。切替手順・事前準備・ロールバックは `docs/EMAIL_SWITCHOVER_RUNBOOK.md` を参照してください。

### 会員番号の付番

登録が完了すると、最小の空き番号が3桁ゼロ埋め形式（例: 001、002）で付番されます。削除された会員の番号は、次の登録者に再利用されます（詳細は「11. 会員の休眠・削除について」参照）。

### 表示名のフォーマット

参加者一覧では、名前は次の形式で表示されます。

- ニックネームあり: `姓 名(ニックネーム)`
- ニックネームなし: `姓 名`

ニックネームが未登録の場合、空の `()` は表示されません。

登録後も、ユーザ画面右上のヘッダーからニックネームを変更できます。ニックネームを変更すると、参加中・キャンセル待ち中のイベントの本人表示にも反映されます。ただし、以前に友達として追加した参加者の表示名（`友達名(◯◯の友達)` の ◯◯ 部分）には反映されません。

## 4. イベント一覧仕様

参加者側トップページには、公開中のイベントが表示されます。

表示されるイベント:

- 受付中イベント（status = accepting）
- 締切済みイベント（status = closed）

表示されないイベント:

- 下書きイベント（status = draft）
- アーカイブ済みイベント（status = archived）

イベント一覧からイベントをタップすると、イベント詳細画面へ移動します。

一覧上の場所表示からGoogle Mapへ直接飛ぶ機能はありません。Google Mapへのリンクはイベント詳細画面内にだけ表示されます。

一覧画面には「申請受付中」または「締め切り済み」のバッジが表示されます。自分が参加中の場合は「✅ 申請済み」、キャンセル待ちの場合は「⏳ 待機中」のバッジも合わせて表示されます。

### アーカイブについて

終了したイベントは削除せず、status を archived に変更して保存します（処理の詳細は「10. 管理者画面仕様」と「12. 運用時の注意点」参照）。アーカイブ済みイベントは、参加者画面・参加者公開ビューから自動的に非表示になります。アーカイブされたイベントの参加者名（参加履歴）はデータベースに保持されます。管理者はアーカイブ済みイベントも閲覧できます。

## 5. イベント詳細仕様

イベント詳細画面では、次の内容を確認できます。

- イベント名
- 開始日時・終了日時（同日の場合は開始時刻〜終了時刻のみ表示）
- 場所（Google Mapリンクが設定されている場合はリンク表示）
- 「申請受付中」または「締め切り済み」のステータスバッジ
- 参加申請フォーム
- 参加者一覧（スロット番号順）
- キャンセル待ち一覧（該当者がいる場合のみ表示）

参加者一覧はリアルタイムで更新されます。別のユーザが参加申請した場合は追加され、キャンセルした場合は一覧から消えます（15秒ポーリングでも同期）。

自分が参加中の場合は「✅ 参加中（N番）」、キャンセル待ちの場合は「⏳ キャンセル待ち（待N番）」と表示されます。自分のエントリは青色でハイライトされます。

ゲスト（友達）のエントリには「臨時ID: XXXXXX」が表示されます。

## 6. 参加申請仕様

ログイン済みの参加者は、イベント詳細画面から参加申請できます。

参加申請できる条件:

- イベントが受付中である（status = accepting）
- 定員に空きがある
- 本人が同じイベントに未参加である（active または waitlist ステータスでない）

参加申請できない条件:

- イベントが締切済みである（status = closed）
- 定員に達している
- すでに本人が参加済みである

**日付や時刻による申請制限はありません。** 締切のトリガーは定員到達か手動締切のみです（詳細は「7. 定員と締切の仕様」参照）。

定員到達時はキャンセル待ちにはなりません。追加申請は受け付けず、イベントは締切済みになります。

## 7. 定員と締切の仕様

デフォルト値は次の通りです。

- 定員上限: 35名
- 再開閾値: 30名

### 定員に達した場合

参加者が35名に達すると、イベントは自動で締切済みになります。

締切後は、参加者がキャンセルしても、参加者数が30名を下回るまでは追加の参加申請はできません。

例:

- 35名: 締切済み
- 34名: 締切済み（定員超過による締切後は閾値未満になるまで再開しない）
- 30名: 締切済み
- 29名: 受付再開（max_participants が 30 にリセットされる）

受付再開時は定員上限が閾値（30名）に引き下げられます。その後さらに参加者が増えて30名に達すると、再び締切済みになります。

### 日付・時刻による自動締切はありません

現在のシステムでは、日付や時刻によってイベントを自動で締め切る仕組みはありません。締切は「定員到達」か「運営による手動締切」のみです。

### 運営が手動で締め切った場合

運営が管理者画面から手動で締め切ったイベント（is_manual_close = true）は、参加者がキャンセルしても勝手に受付再開しません。

再開したい場合は、運営が管理者画面から明示的に再開します。

### 下書きから公開への自動昇格

公開予約日時（publishes_at）を設定した下書きイベントは、その日時を過ぎると自動で受付中に昇格します。Supabase の pg_cron が毎分昇格処理を実行するほか、次のいずれかのタイミングでもサーバ側で昇格チェックが走ります。

- 会員がイベント一覧・イベント詳細ページを開いたとき
- 会員が参加申請・キャンセルを行ったとき
- 管理者がイベント一覧を開いたとき

## 8. キャンセル仕様

参加者本人は、自分の参加をキャンセルできます。

キャンセルは、イベントが締切済みでも可能です。

キャンセルボタンを押すと確認ダイアログが表示されます。

表示文言は状況によって変わります。

### 通常時

イベントが受付中の場合、またはキャンセル後に参加者数が閾値を下回らない場合（自動再開が起きない場合）は、通常の確認文言を表示します。

例:

`参加をキャンセルしてもよろしいですか？`

### 締切中で、キャンセル後も再開しない場合

イベントが締切済みで、自分がキャンセルした後もアクティブ参加者数が閾値（30名）以上のままになる場合は、追加申請がまだ再開されないことを表示します。

例:

`参加者数が30人を下回るまで追加の参加申請はできません。キャンセルしてもよろしいですか？`

## 9. 友達追加仕様

参加者は、イベントごとに友達を代理で追加できます。

友達追加の特徴:

- 自分が参加していなくても、友達だけ追加できます
- 追加できる人数は固定ではなく、その時点の定員空き枠の数まで追加できます（追加のたびに残枠が減ります）
- 入力欄はプラスボタンで増やします
- 友達を追加すると、そのイベントだけで使う臨時IDが発行されます
- 友達は参加者一覧に表示されます
- 友達の表示名は `友達の名前(紹介者の姓の友達)` の形式になります（例: 田中(山田の友達)）
- 自分が追加した友達は、自分でキャンセルできます
- 自分の参加をキャンセルしても、追加済みの友達は自動では消えません
- 自分が参加をキャンセルした後でも、追加済みの友達を管理できます

友達追加できる条件:

- イベントが受付中である（status = accepting）
- 定員に空きがある

友達追加できない条件:

- イベントが締切済みである
- 定員に空きがない

**日付や時刻による制限はありません。** 受付中かつ空き枠がある限り、いつでも友達を追加できます。

## 10. 管理者画面仕様

管理者画面では、運営がイベントを管理できます。管理者は `/admin` からパスワードでログインします。ログインに連続で失敗するとレート制限がかかるため、運用上は正しいパスワードを確実に控えておいてください。

### イベント作成・編集項目

- イベント名
- 開始日時
- 終了日時
- 場所
- Google Maps URL
- 公開予約日時（任意。下書き保存時のみ意味を持ち、設定した日時を過ぎると自動で受付中に昇格します）
- 定員上限（初期値 35名）
- 再開閾値（初期値 30名）
- ステータス

イベントは「下書き保存」または「即時公開」で作成できます。

### ステータス

イベントには次の状態があります。

- 下書き（draft）: 参加者側に表示されない。公開予約日時に達すると自動で受付中に昇格する
- 受付中（accepting）: 参加申請を受け付けている
- 締切済み（closed）: 参加申請を受け付けていない（キャンセルは可能）
- アーカイブ済み（archived）: 終了後の非表示状態。参加者画面には表示されず、管理者のみ閲覧できる

ステータスはトグル操作で切り替えられます（下書き→公開、受付中→締切、締切→受付中など）。締切に切り替えたイベントは手動締切（is_manual_close = true）として扱われ、自動再開しません。受付中や下書きに戻すと、この手動締切フラグは解除されます。

管理者画面では、通常のイベント一覧とアーカイブ済みイベント一覧を切り替えて閲覧できます。

### 参加者管理

運営はイベントごとの参加者一覧を確認できます。アーカイブ済みイベントの参加者一覧も閲覧できます。

必要に応じて、参加者を強制キャンセルできます。

イベント自体を削除すると、紐づく参加者データもすべて削除されます。終了済みイベントは原則として削除ではなくアーカイブ化で保存します。

## 11. 会員の休眠・削除について

会員の `last_accessed_at`（最終アクセス日時）は、ログイン時に必ず更新されます。ログイン後のページ閲覧では、データベースへの書き込みを減らすため、前回の更新から24時間以上経過している場合にのみ更新されます（最大24時間のずれが生じますが、休眠判定は365日基準のため実質的な影響はありません）。365日間アクセスがなかった会員は、毎日実行される cron 処理によって自動的に削除されます。

削除された会員が参加していたイベントの参加者名はデータベースに残ります（participants.member_id が null になり、名前のみ保持）。これにより、過去の参加履歴（誰が参加したか）は失われません。

削除された会員番号は、次の登録者に再利用されます（最小の空き番号を優先して付番）。

## 12. 運用時の注意点

- 定員上限と再開閾値は、原則として `35名 / 30名` を使います
- 日付・時刻による自動締切はありません。締切は「定員到達」か「手動締切」のみです
- 手動で締め切ったイベントを再開したい場合は、運営が管理者画面から再開します
- 公開予約日時を設定した下書きは、日時到達で自動的に受付中へ昇格します（毎分のサーバ処理に加え、会員のページ閲覧・参加申請・キャンセル・管理者一覧表示時のチェック）
- 終了したイベントは、毎日の自動処理によりアーカイブ化されます。アーカイブ済みイベントはユーザー画面から非表示になりますが、データは削除されず、管理者は閲覧できます
- 365日アクセスのない会員は毎日の自動処理で削除されます。削除しても参加履歴の名前は残り、会員番号は再利用されます
- 参加者本人のキャンセルは、締切済みイベントでも可能です
- 友達は、追加した本人が管理します
- 友達の臨時IDはイベントごとの管理用で、恒久的な会員IDではありません
- 友達の表示名の「◯◯の友達」部分は、追加時点の招待者の姓が埋め込まれます。招待者がニックネームを変更しても、この表示は変わりません
- 確認メールの送信は1時間あたり最大2通の制限があります。短時間に複数回の登録テストを行う場合は注意してください
- 本格運用前には、テスト用ユーザや不要なイベントを削除します

## 14. 運営展開用デモイベント

運営説明用として、`scripts/qa-create-rollout-demo-events.mjs` で次のデモイベントを作成できます。スクリプトは実行時の「今日」（JST）を基準に各イベントの開催日を設定し、既存の `【運営展開用】` プレフィックス付きイベントを事前にすべて削除してから作り直します。本番環境への実行は、明示的な確認用フラグ（`ALLOW_PRODUCTION_QA=1`）を設定しない限り拒否されます。

> 以下は説明用の構成例です。開催日はスクリプト実行日からの相対日数で決まります。全イベント共通で 定員上限 35名 / 再開閾値 30名、締切日時は使用しません。

### 受付中・参加/友達追加デモ（受付中）

見るポイント:

- 通常の参加申請
- 友達追加
- 友達の臨時ID表示
- 参加者一覧の表示（参加者3名＋ゲスト友達1名で開始。いずれもデモ用データで、会員アカウントには紐付きません）
- ニックネーム付き表示名のデモ（先頭の参加者は「佐藤 太郎(たろちゃん)」と表示され、`姓 名(ニックネーム)` 形式の実例になります）

### 定員35名締切デモ（締切済み）

見るポイント:

- 定員到達時の締切状態
- 追加申請できない状態
- 参加者数 35 / 35 の表示

### 閾値30名・再開待ちデモ（締切済み）

見るポイント:

- 参加者がちょうど30名（閾値）の状態
- 30名ではまだ再開しない状態
- 30名を下回ると受付再開する仕様の説明

### 手動締切デモ（締切済み）

見るポイント:

- 参加者が少ない（2名）状態での手動締切
- キャンセルしても自動再開しないこと

### 下書き固定デモ（下書き）

見るポイント:

- 下書きイベント
- 参加者側にはまだ表示されない状態
- 公開予約日時を設定していないため自動公開されないこと

### 自動公開予定デモ（下書き）

見るポイント:

- 公開予約日時を設定した下書きイベント
- 設定日時に達すると自動で受付中へ昇格すること
